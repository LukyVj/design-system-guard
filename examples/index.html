Design System Guard

A Claude Code plugin for frontend platform engineers who own design-system adoption across many React teams. It catches the same recurring PR issues — hardcoded colors, arbitrary Tailwind values, inline styles, and one-off components — and suggests safe migrations to approved tokens.

Below: Button and Input fixtures, before and after. Both render live; the code snippets match the actual files in examples/bad/ and examples/good/.

Button — drift vs approved

✗ Drift — bad/Button.tsx

Live render

Code

<button
  className="btn rounded-[9px] bg-[#121212]
    px-[13px] py-[7px] text-[15px] text-white
    hover:bg-[#ff00aa] disabled:opacity-50"
  style={{ boxShadow: "0 0 0 1px #333333" }}
>
  {children}
</button>

Flagged by ds-check.mjs

  • Hardcoded hex — #121212, #ff00aa, #333333
  • Arbitrary Tailwind color — bg-[#121212], hover:bg-[#ff00aa]
  • Arbitrary Tailwind size — rounded-[9px], px-[13px], py-[7px], text-[15px]
  • Inline style — boxShadow
  • One-off component — raw <button> instead of approved Button
✓ Approved — good/Button.tsx

Live render

Code

import { Button as DSButton }
  from "@acme/design-system";

<DSButton
  variant="primary"
  size="md"
  onClick={onClick}
  disabled={disabled}
>
  {children}
</DSButton>

Why it passes

  • No hardcoded colors, arbitrary values, or inline styles
  • Uses approved component via variant / size props
  • Color, radius, spacing, shadow live in the design system, not the call site

Input — drift vs approved

✗ Drift — bad/input.tsx

Live render

Code

<input
  className="input rounded-[9px] bg-[#121212]
    px-[13px] py-[7px] text-[15px] text-white
    hover:bg-[#ff00aa] disabled:opacity-50"
  type="text"
  value={value}
  onChange={(e) => onChange(e.target.value)}
/>

Flagged by ds-check.mjs

  • Hardcoded hex — #121212, #ff00aa
  • Arbitrary Tailwind color — bg-[#121212], hover:bg-[#ff00aa]
  • Arbitrary Tailwind size — rounded-[9px], px-[13px], py-[7px], text-[15px]
  • Palette color — text-white
  • One-off component — raw <input> instead of approved Input
✓ Approved — good/Input.tsx

Live render

Code

import { Input as DSInput }
  from "@acme/design-system";

<DSInput
  value={value}
  onChange={onChange}
/>

Why it passes

  • No hardcoded colors or arbitrary Tailwind values at the call site
  • Uses approved Input from the design-system package
  • Border, focus, spacing, typography owned by the design system

What the plugin does

Skill
/design-system-guard:audit-component <file> — audit a component for design-system drift
Agent
design-system-migrator — suggests and applies minimal diffs to approved tokens/components
Hook
PostToolUse after Write/Edit — warns when a new violation appears
Checker
bin/ds-check.mjs — dependency-free scanner for hex colors, arbitrary Tailwind values, inline styles
Tokens
examples/tokens.json — the approved palette, type scale, spacing, radius, shadows

Try it

# Button — expect warnings on bad, clean on good
node bin/ds-check.mjs examples/bad/Button.tsx
node bin/ds-check.mjs examples/good/Button.tsx

# Input — expect warnings on bad, clean on good
node bin/ds-check.mjs examples/bad/input.tsx
node bin/ds-check.mjs examples/good/Input.tsx
